==================================================
1. CURRENT ARCHITECTURE
==================================================

The project is currently organized like this:

- sim.py
  - Runs Genesis simulation
  - Computes robot state
  - Sends feedback to host
  - Draws debug markers

- host.py
  - ZMQ hub between ctrl / sim / external perception app
  - Drives hardware when hardware is enabled
  - Handles current limiting / safety fault
  - Converts perception camera-frame coordinates into world coordinates
  - Broadcasts debug markers and state

- ctrl.py
  - IMPORTANT: currently restored to launcher-only
  - It should stay thin
  - Real logic is in engine/controller/* and ui/*

- engine/ik.py
  - Public IK facade
  - solve_then_align / tweak helpers

- engine/iklib/*
  - Internal IK implementation
  - solver.py
  - aligner.py
  - tweaker.py
  - kinematics.py

- engine/controller/*
  - Non-UI application/controller layer
  - state.py
  - client.py
  - actions.py
  - perception.py

- ui/*
  - imgui-specific UI layer
  - ui/control_panel.py
  - ui/panels/hardware.py
  - ui/panels/control_4dof.py
  - ui/panels/ik.py
  - ui/panels/sag.py

- addons/autonomous_pick_place_app
  - External companion perception app
  - Uses camera / detector
  - Sends object_camera coordinates to host

- addons/perception_bridge/*
  - hand-eye and transform helpers used by host

==================================================

At one point another Codex session overwrote ctrl.py with an old monolithic version
(~1500 lines, containing HostState, ControlClient, ControlService, imgui code all
inside ctrl.py).

This broke the new split architecture because:
- engine/controller/* still existed

This has already been repaired.

Current ctrl.py is launcher-only again and wires together:
- engine.controller.ControlClient
- engine.controller.ControlService
- engine.controller.PanelState
- ui.control_panel.ControlPanel

If ctrl.py ever becomes a monolith again, that is a regression.


==================================================
3. IK / ALIGN / TWEAK STATUS
==================================================

Current intended roles:

- Solve IK
  - solve position
  - then align direction if possible

- Align
  - open-loop alignment step
  - conceptually:
    1. find a 4DOF position solution
    2. lock roll
    3. search 3DOF feasible space for better direction

- Tweak
  - closed-loop fine adjustment
  - position-priority
  - direction is secondary / nullspace improvement
  - uses actual_tip_xyz + actual_tip_dir when feedback exists
  - supports accept/reject, rollback, step shrink

Important note:
- Tweak is better than before, but still not fully proven on real hardware.
- In simulation it is plausible.
- On real hardware it still needs validation.

==================================================
5. SAFETY / CURRENT LIMITING
==================================================

Current protection policy in host.py:

- yellow zone starts at 1800 mA
- red line is 2500 mA

Behavior:
- yellow zone:
  - command scaling decreases linearly
- red zone:
  - safety trip / fault

host.py logs:
- yellow zone entry
- yellow zone clear
- red zone trip

Hardware panel already shows:
- safety_fault
- motor_currents_ma

This was important because the geared mechanism is not backdrivable and motors
can die if pushed into load too aggressively.


==================================================
6. 4-DOF OFFSET / CALIBRATION FEATURE
==================================================

This is one of the biggest additions in this session.

Files:
- engine/controller/state.py
- engine/controller/actions.py
- ui/control_panel.py
- ui/panels/control_4dof.py

Feature:
- Each axis has offset input + Apply button
  - linear
  - roll
  - seg1
  - seg2

Meaning:
- UI keeps showing vanilla display coordinates
- actual sent value is display + offset

Example:
- seg1 offset = -10
- UI slider 0
- actual sent seg1 = -10

Calibration:
- only for seg1 and seg2
- slowly move toward end
- detect current rise
- slowly release
- detect first released point
- write that value into offset
- apply automatically

Current calibration constants in engine/controller/actions.py:
- threshold = 1400 mA
- abort = 2000 mA
- EMA based

UI shows calibration status message.


==================================================
7. EXTEND ARM BUTTON
==================================================

There used to be a Reset/Home naming issue.

The command is not a logical reset and not a universally agreed "home".
Its actual purpose is to send the arm to a predefined extended pose.

Current button label:
- "Extend Arm"

Current behavior:
- sends actual values corresponding to:
  - linear = 15
  - roll = 180
  - seg1 = 180
  - seg2 = 180

Important:
- Extend Arm intentionally SKIPS offsets
- this guarantees it always goes to the same actual physical pose
- normal slider operations still apply offsets

==================================================
12. PERCEPTION / AUTONOMOUS_PICK_PLACE_APP
==================================================

The design philosophy is:
- keep perception outside core Elesim
- use a companion app

Current structure:
- addons/autonomous_pick_place_app
  - computes object_camera in camera optical frame
  - sends it to host

- host.py
  - converts object_camera -> object_world
  - broadcasts debug markers to sim
  - returns object_world inside ACK back to external app

- sim.py
  - only renders markers

So the path is:
- camera app -> host (5555) -> sim markers

Additional improvement:
- host ACK now includes object_world
- autonomous_pick_place_app prints both camera-frame and world-frame results

Files involved:
- host.py
- addons/autonomous_pick_place_app/elesim_bridge/host_client.py
- addons/autonomous_pick_place_app/main.py

Also:
- host logs perception success with camera/world coordinates
